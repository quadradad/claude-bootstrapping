# Project Configuration

## Development Philosophy

- **TDD:** Write tests first. Red → green → refactor. No exceptions.
- **Issue-Driven:** All work tracked via the project's task tracker.
- **Plan Before Building:** Plan mode for 3+ step tasks or architectural decisions. If implementation hits unexpected complexity, stop and re-plan.
- **Fix Bugs Autonomously:** Given logs, errors, or failing tests — diagnose and resolve directly. Zero context switches from the user.
- **Challenge Your Work:** For non-trivial changes, pause and ask "is there a more elegant way?" before presenting.
- No credentials, secrets, or .env files committed.

## Subagent Strategy

- Offload research, exploration, and parallel analysis to subagents
- One focused task per subagent — results summarized back to main thread

## Session Start

1. Review `.claude/lessons.md` for patterns relevant to current work. If >40 entries, flag for pruning.
2. Assess state: run `/triage` or check the task tracker
3. Decide next action: continue interrupted work, pick next task, or plan new work

## Continuous Improvement

Maintain `.claude/lessons.md` with patterns from corrections and reviews. See `agent_docs/self-improvement.md` for format and triggers.

## Issue Tracker

**Tool:** GitHub CLI (`gh`) | **Format:** `#NN` | **Smart close:** `Closes #NN` | **Deps:** `- Blocked by: #NN — reason`
For issue format specs, see `agent_docs/issue-conventions.md`. For CLI operations, see `agent_docs/issue-tracker-ops.md`.

## Commit Conventions

Conventional commits: `type(scope): description` with smart close syntax.
Types: `feat`, `fix`, `refactor`, `docs`, `discovery`, `design`, `infra`.
Never force-push or rewrite history on shared branches.

## PR Workflow

- Every change gets a PR. Branch naming: `{issue-number}-{slug}`
- Feature branches target release branch (if exists), otherwise `main`
- Use `/review-pr` before merging. One issue per branch.

## Pre-existing Failures

If a test file you did NOT modify is failing, create an issue and continue. Do not silently work around it.
